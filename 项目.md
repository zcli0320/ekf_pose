# EKF 融合与 SLAM/VO 初始化实现说明

更新时间：2026-05-12

本文记录当前工程中 EKF 融合节点、SLAM/VO 引导初始化、odom 中断退化与后续运动中重新初始化的底层实现逻辑。说明以当前代码为准，主要对应 `src/ekf_node_vio_timesync_with_acc_pub.cpp`、`scripts/vo_gnss_imu_guidance.py` 和 `launch/vo_guided_ekf.launch`。

## 1. 状态量、输入量、观测量和协方差对应关系

当前 EKF 是误差状态形式。名义状态为：

```text
X = [p, q, v, bg, ba]
```

其中：

- `p`：位置，3 维。
- `q`：姿态四元数，4 维。
- `v`：速度，3 维。
- `bg`：陀螺零偏，3 维。
- `ba`：加速度计零偏，3 维。

名义状态共 16 维。误差状态为：

```text
dx = [dp, dtheta, dv, dbg, dba]
```

误差状态共 15 维，协方差矩阵 `P` 也是 15x15。这里姿态误差使用 3 维小角度 `dtheta`，不是四元数 4 维误差。

传感器和矩阵对应关系如下：

- IMU 输入：`gyro, acc`，用于预测，过程噪声为 6x6 的 `Qt`。
- odom 观测：位置 + 姿态，形成 6 维 residual，观测噪声为 `Rt`。
- GNSS 观测：`NavSatFix` 转 ENU 后作为 3D 位置观测，使用独立 GNSS `R`。
- odom 丢失退化时，可选使用 GNSS 连续位置差分形成速度伪观测，更新位置和速度状态。

## 2. 程序启动后的总体流程

使用 `launch/vo_guided_ekf.launch` 时，系统由两部分组成：

1. `vo_gnss_imu_guidance.py`
   - 订阅 raw SLAM/VO odom、GNSS 和 IMU。
   - 将未对齐的 raw SLAM/VO 轨迹估计成带尺度、yaw 和平移的 `/ekf/guided_vo_odom`。

2. `ekf_lidar.launch`
   - 启动 EKF 主节点。
   - 订阅 IMU、GNSS 和 `/ekf/guided_vo_odom`。
   - 输出 `/ekf/ekf_odom`、`/ekf/cam_ekf_odom`、`/ekf/input_path`、`/ekf/ekf_path`、`/ekf/gnss_path` 等结果。

关键点是：raw SLAM/VO 不直接进入 EKF。它必须先经过引导初始化节点，生成 guided odom 后，才作为 EKF 的 odom 观测输入。

## 3. EKF 主节点运行流程

EKF 主节点启动后会读取噪声参数、GNSS 健康管理参数、odom lost 参数、路径发布参数等，然后进入回调驱动流程。

### 3.1 IMU 预测

IMU 是 EKF 的预测输入。每来一帧 IMU，节点根据时间间隔 `dt` 传播：

- 位置 `p`
- 姿态四元数 `q`
- 速度 `v`
- 协方差 `P`

如果 IMU 时间戳乱序，即 `dt <= 0`，当前逻辑会忽略该帧，避免状态被异常时间戳污染。

### 3.2 odom 初始化与更新

如果 EKF 尚未初始化，第一帧有效 odom 可以初始化位置、姿态和速度。后续 odom 作为位置 + 姿态观测进入 EKF 更新。

当前 odom 更新带有 IMU 时间同步逻辑：

- EKF 缓存一段 IMU 传播历史。
- odom 到达时，回到对应 IMU 时间附近做更新。
- 更新后再用后续 IMU 重新传播到当前时刻。
- 如果 odom 时间戳比当前 IMU buffer 更新，先进入 pending 队列，等待 IMU buffer 追上后再处理。

这样可以减少 rosbag 回放、消息成批到达或传感器异步带来的状态跳变。

### 3.3 GNSS 冷启动与位置更新

如果系统尚未收到 odom，但 GNSS 已经有效，EKF 可以执行 GNSS 冷启动：

- 第一帧有效 GNSS 建立 ENU 原点。
- EKF 位置初始化为 GNSS ENU 局部坐标。
- 姿态初始化为单位四元数。
- yaw 在物理上仍不可观，只是给出默认初值。

后续 GNSS 更新前，节点会将 GNSS ENU 与当前 odom/map 坐标系做对齐。健康 odom 存在时，如果 GNSS yaw 对齐尚未完成，默认不直接用 GNSS 拉动 EKF 位置。

## 4. SLAM/VO 初始化的实际实现

当前 SLAM/VO 初始化由 `scripts/vo_gnss_imu_guidance.py` 完成。默认输入为：

```text
raw_vo_topic = /Odometry
imu_topic    = /mavros/imu/data
gnss_topic   = /mavros/global_position/global
output       = /ekf/guided_vo_odom
```

该节点估计 raw SLAM/VO 到 GNSS/ENU 坐标系的 2D 相似变换：

```text
guided_xy = scale * R_yaw * raw_vo_xy + translation_xy
guided_z  = scale * raw_vo_z + translation_z
```

估计参数包括：

- `scale`：SLAM/VO 尺度。
- `yaw`：SLAM/VO 坐标系相对 GNSS/ENU 的水平航向偏差。
- `translation_x, translation_y`：水平平移。
- `translation_z`：高度偏移。
- `residual_mean, residual_max`：拟合残差。

真正参与 `scale/yaw/translation` 拟合的是 raw SLAM/VO 位置序列和 GNSS ENU 位置序列。IMU 当前主要作为门控条件，用于确认最近存在 IMU 数据；它没有直接参与尺度和 yaw 的数值拟合。

因此，从代码实现角度看，当前初始化逻辑应描述为：

```text
raw SLAM/VO + GNSS 约束估计，IMU 做可用性门控
```

如果 `/Odometry` 外部来源本身是 VIO，那么输入物理上可以称为 VIO odom；但当前引导脚本没有读取 VIO 内部状态，也没有使用 VIO 的惯性约束信息，所以算法层面仍按 raw SLAM/VO odom 处理。

## 5. SLAM/VO 初始化触发条件

初始化不是 SLAM/VO 一启动就触发，而是必须通过一组短窗口门控。

### 5.1 时间同步与 IMU 可用性

GNSS 到来后，节点会在 raw SLAM/VO buffer 中寻找时间最近的 odom 样本。默认最大配对时间差：

```text
max_pair_dt = 0.08 s
```

如果 `require_imu=true`，还要求最近一段时间内有 IMU：

```text
imu_timeout = 0.5 s
```

这一步的作用是避免传感器数据不完整或时序严重错位时误初始化。

### 5.2 配对样本数足够

默认至少需要：

```text
min_pairs = 8
```

少于该数量时不做相似变换拟合，状态为 `not_enough_pairs`。

### 5.3 水平运动距离足够

默认要求：

```text
min_motion = 10.0 m
```

这是关键门槛。当前要估计的是 `scale + yaw + translation`。如果运动距离过短，GNSS 噪声、SLAM/VO 抖动和时间同步误差会占主导，yaw 和 scale 不可稳定观测。

例如只移动 0.5 m 时，0.3 m 级别 GNSS 噪声足以显著改变方向估计；移动 10 m 后，同样噪声占比降低，方向和尺度估计更可靠。

### 5.4 水平速度足够

默认要求：

```text
min_speed = 5.0 m/s
```

该条件不是数学上绝对必须，但在工程上用于排除低速误初始化。低速时：

- GNSS 相邻点差分方向容易被噪声污染。
- SLAM/VO 初始轨迹可能只是局部抖动。
- yaw 和 scale 估计容易漂。

如果使用纯 VO，该条件建议保留。如果后期使用高质量 VIO，并且 VIO 已经提供可靠尺度和姿态，可以适当降低阈值，但不建议直接取消，应增加 VIO tracking status、feature count、协方差、短窗口残差等质量判断。

### 5.5 高度变化不能太大

默认要求：

```text
max_vertical_motion = 2.0 m
```

当前拟合模型是 2D 水平相似变换，不是完整 3D Sim(3)。它主要估计水平 `scale + yaw + translation`，高度只用平均偏移处理。因此初始化窗口应接近水平飞行。

如果在垂直起降或大幅爬升下降中初始化，当前模型可能把高度变化、GNSS 高度噪声和 SLAM/VO 尺度误差混在一起，导致错误 scale 或高度偏移。

### 5.6 速度近似匀速

默认要求：

```text
uniform_speed_max_cv = 0.35
```

程序计算 GNSS 相邻段速度的变异系数，排除强加速、急减速和复杂机动窗口。虽然相似变换拟合本身不严格要求匀速，但近似匀速可以降低时间同步误差和局部转弯对 yaw/scale 的影响。

对于运动中重新初始化，这个条件尤其重要。重新初始化不应发生在急转弯、避障或剧烈加减速窗口中，应等待更稳定的水平匀速段。

### 5.7 scale、残差和稳定性检查

拟合成功后，还需要通过：

```text
min_scale = 0.05
max_scale = 20.0
max_residual = 1.0 m
ready_frames = 4
scale_stability = 0.05
yaw_stability = 0.05 rad
```

含义是：

- scale 不能过大或过小。
- 拟合后 VO-GNSS 残差不能超过阈值。
- 不是一次拟合成功就 ready，而是连续多次拟合出的 scale 和 yaw 稳定后才发布 `/ekf/guided_vo_odom`。

## 6. VO 与 VIO 的选择

后期如果需要处理“运动过程中 odom 中断并重新初始化”，推荐主方案使用 VIO，而不是纯 VO。

原因如下：

- VIO 自身融合相机和 IMU，通常能输出有尺度、姿态连续性更好的 odom。
- 无人机运动中重新初始化时，VIO 可以利用 IMU 约束短时间姿态、速度和尺度，恢复速度通常优于纯 VO。
- 纯 VO 在重新启动后容易存在尺度、yaw 和平移不确定，必须经过 GNSS/IMU 引导窗口重新估计后才能接入 EKF。

但 VIO 也有代价：EKF 已经使用 IMU 做预测，而 VIO 输出内部也使用 IMU，二者误差并非完全独立。工程上可以接受，但需要保守处理：

- VIO 作为 odom 观测源，而不是当作真值。
- odom 观测噪声 `Rt` 不应设置过小。
- VIO reset 或 relocalization 后不能立即强融合，应先进入恢复检查。
- 如果 VIO 提供 covariance、tracking status、feature count、reset id，应接入健康管理。

纯 VO 更适合作为受限条件下的备选方案。若只能使用纯 VO，则必须依赖当前这种 GNSS/IMU 引导初始化逻辑，并且只在满足运动窗口条件时接入 EKF。

## 7. odom 中断后的当前处理能力

如果 SLAM/VO 首次初始化后 `/ekf/guided_vo_odom` 中断，EKF 会通过 `odom_loss_timeout` 检测 odom lost。默认：

```text
odom_loss_timeout = 1.0 s
```

进入 odom lost 后：

- IMU 继续预测。
- GNSS 继续作为位置观测约束。
- 如果 `enable_gnss_velocity_when_odom_lost=true`，使用连续 GNSS 差分速度形成速度伪观测。
- 展示默认配置中该选项为 `false`，避免 rosbag 回放消息成批到达时误触发 GNSS 速度伪观测并拉偏轨迹。

如果 odom 后续恢复：

- 若恢复后的 odom 与历史 frame 连续，EKF 可以继续融合。
- 若恢复后出现明显跳变，当前逻辑会尝试 odom frame realign。
- 如果 SLAM/VO 重新初始化导致 frame 改变，realign 有助于保持 EKF 输出连续。

当前不足是：EKF 主节点还没有完整显式的 `SLAM_LOST -> SLAM_REINIT_PENDING -> SLAM_REALIGNING -> SLAM_RECOVERED` 状态机。恢复时主要依赖 odom 跳变检测、残差、自适应协方差和 realign 逻辑。

## 8. 后期运动中重新初始化建议

为了更稳地处理运动过程中 odom 中断和重新初始化，建议后续增加显式恢复流程：

```text
ODOM_HEALTHY
ODOM_LOST
ODOM_REINIT_PENDING
ODOM_REALIGNING
ODOM_RECOVERED
```

恢复判断建议同时使用：

- odom 中断持续时间。
- IMU+GNSS 预测位置与新 odom 的短窗口残差。
- 新旧 odom frame 的 yaw 和 translation 差。
- VIO/SLAM tracking status。
- feature count 或匹配质量。
- odom covariance。
- IMU yaw rate 或 VO angular velocity，用于拒绝转弯/强机动窗口。
- GNSS 健康评分和 NIS 状态。

建议策略：

- 低速、悬停、纯垂直运动：禁止或延迟重新初始化。
- 高速水平近匀速：允许初始化和对齐。
- 转弯、急加速、避障：等待更稳定窗口。
- VIO 输入且质量高：可以适当降低速度和运动距离阈值。
- 纯 VO 输入：保持严格阈值，避免错误尺度和 yaw 污染 EKF。

## 9. 当前结论

当前工程已经具备：

- IMU+odom+GNSS 误差状态 EKF。
- GNSS 冷启动。
- GNSS ENU 与 odom/map 的 yaw+translation 对齐。
- odom 跳变检测与 frame realign。
- GNSS NIS/健康评分/运动一致性检测。
- odom lost 后 IMU+GNSS 退化运行。
- 受限条件下的 SLAM/VO 运动中引导初始化。

但对于后期目标“odom 中断后在运动中重新初始化并稳定恢复”，当前实现还需要增强显式状态机和恢复窗口质量判断。推荐后期主输入优先使用 VIO，纯 VO 作为受限条件下的备选输入。
