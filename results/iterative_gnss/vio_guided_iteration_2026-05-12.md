# VIO Guided Initialization Iteration - 2026-05-12

## Scope

This iteration keeps the EKF state definition unchanged and adds a VIO-guided initialization path:

- VIO/odom remains a `nav_msgs/Odometry` observation.
- EKF state remains `p, q, v, bg, ba`; error state remains 15D.
- IMU remains the prediction input with 6D process covariance `Qt`.
- Odom/VIO remains a 6D pose residual with covariance `R_odom`.

## Changes Validated

- Added wider VIO/GNSS alignment windows using sparse GNSS reference samples.
- Disabled unsafe initial translation-only alignment by default.
- Kept translation-only alignment only as a weak recovery option after yaw has already been estimated.
- Made `/mavros/odometry/in` the default raw VIO/odom source for VIO-guided initialization.
- Added GNSS fallback-only mode for the VIO launch: GNSS is recorded for reference while odom is healthy, and only updates EKF when odom is lost or EKF has not initialized.
- Enabled odom/VIO message covariance in the VIO launch so the guided odom covariance can weaken low-confidence alignment.
- Tuned GNSS cold start delay to 8s in VIO launch. A 5s delay was rejected because it can preempt VIO on `gps_fusion.bag`.
- Added benchmark support for custom launch files.

## Interpretation

- `all_gps.bag` and `gps_fusion.bag` validate the intended default path: IMU + VIO/odom initializes EKF before GNSS cold start, no reset occurs, and GNSS does not directly update EKF while odom is healthy.
- `gps_fusion.bag` contains odom-lost periods. GNSS fallback velocity updates trigger during those periods without causing resets.
- `new_data.bag` does not provide a healthy metric VIO source for this mode. The VIO guidance correctly does not enter READY and EKF falls back to GNSS cold start plus GNSS velocity pseudo-updates.
- The fallback path is functional, but the 2.05 m max step in `new_data.bag` is not considered a VIO initialization success criterion.

Future algorithm iterations must include full bag playback metrics before being considered complete.
