## 六、Layer 4：工程回放与展示验收测试

目标：验证算法不仅在离线指标上通过，也能在答辩/演示使用的 rosbag 回放、慢速播放和 RViz 可视化中稳定表现。该层重点覆盖 2026-05-11 展示 `all_gps.bag` 时发现的问题：红色 SLAM/odom、黄色 GNSS、蓝色实际 odom 观测路径基本一致，但绿色三源融合轨迹出现异常跳变。分析表明，问题主要来自 bag 消息按记录时间成批到达、`header.stamp` 滞后、在线回调顺序和 `odom_lost` 退化逻辑之间的不匹配；其中 GNSS 速度伪观测在展示回放中可能被误触发，进而把 EKF 速度和位置拉偏。

### 6.1 rosbag 回放与 RViz 展示一致性

| 测试 | 做法 | 通过标准 |
|------|------|----------|
| L4-T1 正常速度展示 | `roslaunch ekf ekf_lidar.launch` 后正常播放 `all_gps.bag` | RViz 同时显示红/黄/蓝/绿路径；红黄蓝一致时，绿线不得出现米级非物理跳变；轨迹在 bag 结束后保留 |
| L4-T2 慢速展示 | 使用 `rosbag play --clock -r 0.3 /home/zcl/catkin_ws/src/ekf/all_gps.bag` | 与正常速度播放的轨迹形态一致；不得因为慢速播放改变融合结论 |
| L4-T3 消息成批到达 | 统计 IMU/odom/GNSS 的 `record time - header.stamp`，重点检查开头和 9152s 附近 | 能识别该类 bag 的时序风险；不得把短暂回放延迟误判为真实传感器丢失 |
| L4-T4 展示参数锁定 | 使用展示默认参数，尤其 `enable_gnss_velocity_when_odom_lost=false` | 展示版本不启用 GNSS 速度强制回填；若要验证 odom 丢失退化，必须单独开启并单独记录 |
| L4-T5 轨迹话题一致性 | 对齐 `/ekf/ekf_odom`、`/ekf/cam_ekf_odom`、`/mavros/odometry/in`、`/ekf/gnss_path` | 蓝线应与实际 odom 观测重合；绿色 EKF 与蓝线 P95 偏差、最大 step、GNSS reject/reset 次数均记录入报告 |

### 6.2 2026-05-11 问题复现实验

| 现象 | 复现条件 | 诊断结论 | 验证要求 |
|------|----------|----------|----------|
| 绿线异常但红黄蓝基本一致 | `all_gps.bag` 正常或 0.3 倍速播放 | 输入观测本身不是主因，问题发生在 EKF 状态传播、GNSS 退化分支或可视化发布路径 | 必须同时输出 `ekf_vs_meas_p95`、`ekf_step_max`、`GNSS observation health`、`Odom observation health` 日志 |
| `Odom observation health=LOST` 短暂出现 | bag 开头或消息成批到达区间 | 可能是回放到达顺序造成的伪丢失，不应直接触发强退化策略 | 展示模式下默认不使用 GNSS velocity pseudo-update |
| GNSS 速度伪观测拉偏绿线 | `enable_gnss_velocity_when_odom_lost=true` 且误判 odom lost | GNSS 差分速度会被写入 EKF 速度，可能在健康 odom 实际存在时造成绿色融合轨迹偏离 | 仅在真实 odom 长时间丢失验证中开启；普通三源展示必须关闭 |

### 6.3 无人机高速匀速飞行初始化 SLAM/VO

目的：验证无人机在任务中无法静止等待初始化时，能否利用 GNSS/IMU 提供的短窗口运动约束，在 `>=5 m/s` 近似匀速飞行阶段初始化或引导 SLAM/VO，使其尺度、yaw、平移和高度偏移快速收敛，再作为 odom 观测接入 EKF。

| 测试 | 数据设计 | 通过标准 |
|------|----------|----------|
| L4-SLAM1 匀速直线初始化 | 无人机水平速度 `>=5 m/s`，持续 5-10s 近似匀速，GNSS 健康，SLAM/VO 从无尺度或未对齐状态启动 | 只在速度阈值、方向稳定性和 GNSS 健康同时满足时触发初始化；初始化后 `/ekf/guided_vo_odom` 与 GNSS/odom 方向一致 |
| L4-SLAM2 低速禁止初始化 | 悬停、低速 `<5 m/s` 或频繁加减速 | 不触发 SLAM/VO 尺度和 yaw 初始化，避免低动态下 yaw/尺度不可观导致错误对齐 |
| L4-SLAM3 转弯/机动拒绝 | 速度虽 `>=5 m/s`，但窗口内 yaw rate 较大或速度方向快速变化 | 不使用该窗口初始化，等待更稳定的匀速段 |
| L4-SLAM4 初始化后接入融合 | SLAM/VO 初始化成功后作为 `/ekf/guided_vo_odom` 或等价 odom 输入接入 EKF | EKF 不 reset；绿色融合轨迹连续；SLAM/VO 与 GNSS/odom 的短窗口残差保持在阈值内 |
| L4-SLAM5 初始化失败回退 | GNSS 弱、SLAM 特征不足或速度窗口不满足 | 系统保持 IMU+GNSS 或 IMU+odom 现有模式，不把未初始化 SLAM 当作健康 odom 融合 |

判据建议：

- 速度阈值：GNSS 水平速度或融合速度 `>=5 m/s`。
- 匀速性：窗口内水平速度标准差不超过均值的 `15%`。
- 方向稳定性：窗口内航向变化小于 `5°` 或 yaw rate 小于设定阈值。
- 初始化质量：SLAM/VO 对齐后的水平残差 P95 小于 `1.0 m`，不出现单步 `>0.5 m` 的初始化跳变。
- 工程约束：该初始化逻辑必须有显式状态标志和日志，不能静默把 raw SLAM/VO 直接送入 EKF。

## 九、关键原则

1. **先保证不崩溃，再追求精度。** NaN、奇异矩阵、状态机死锁是最高优先级 bug。
2. **隔离测试优于集成测试。** 一个工况一个工况地验证，不要混在一起跑。
3. **异常注入必须可复现。** 用固定随机种子，每个异常注入都有明确的时间区间和参数。
4. **健康管理比融合公式更关键。** 错误观测被当作健康观测融合，是最危险的失效模式——这也是你自己分析出的结论，验证方案要充分体现这一点。
5. **yaw unobservable 是固有物理限制，不是 bug。** 验证时要区分"算法缺陷"和"传感器配置不足"。
