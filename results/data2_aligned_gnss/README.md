# data2 aligned GNSS bag

This directory contains derived artifacts for `/home/zcl/data2.bag`.

## Root cause

The raw GNSS topic `/mavros/global_position/raw/fix` is valid and frequent, but its
local ENU trajectory is not related to `/mavros/odometry/out` by one stable rigid
transform over the full bag. A full-bag 2D rigid alignment left about 7.07 m P95
horizontal residual, while early samples aligned well. Sliding windows showed the
GNSS-to-odom yaw/offset changing over time, so strong EKF fusion of the raw GNSS
pulls the filter toward an inconsistent frame.

## Generated topic

`data2_with_aligned_gnss.bag` keeps all original messages and adds:

- `/ekf/aligned_gnss/fix`: `sensor_msgs/NavSatFix`, 1711 messages

The topic is generated by locally aligning raw GNSS samples to the odom trajectory
with a sliding 2D rigid fit, then re-encoding the aligned local positions as a
NavSatFix stream for the existing EKF input path. A small odom-guided continuity
constraint limits local window-switching artifacts in the generated GNSS stream.

Generation command:

```bash
python3 dataset_tools/align_gnss_to_odom_bag.py \
  --input-bag /home/zcl/data2.bag \
  --output-bag results/data2_aligned_gnss/data2_with_aligned_gnss.bag \
  --gnss-topic /mavros/global_position/raw/fix \
  --odom-topic /mavros/odometry/out \
  --output-topic /ekf/aligned_gnss/fix \
  --window-s 30 \
  --min-pairs 20 \
  --max-step-correction 0.10 \
  --max-z-step-correction 0.08 \
  --covariance-xy 4.0 \
  --covariance-z 9.0
```

Observed generation residual:

- local fit mean: 0.097 m
- local fit P95: 0.229 m
- aligned GNSS vs odom: mean 0.131 m, P95 0.399 m, max 0.794 m
- aligned GNSS step-vs-odom step error: P95 0.100 m, max 0.100 m
- aligned GNSS z step: P95 0.108 m, max 0.141 m
- aligned GNSS z-vs-odom z-step error: P95 0.080 m, max 0.080 m

## EKF check

Checked with:

```bash
roslaunch ekf ekf_lidar.launch start_rviz:=false \
  odom_primary_topic:=/mavros/odometry/out \
  gnss_topic:=/ekf/aligned_gnss/fix \
  use_gnss:=true \
  enable_gnss_yaw_alignment:=false \
  gnss_require_yaw_alignment_before_update:=false \
  gnss_min_interval:=0.2 \
  gnss_min_cov_xy:=1.0 \
  gnss_min_cov_z:=4.0 \
  gnss_cov_scale:=0.25 \
  position_cov:=0.02
```

EKF logs showed GNSS updates reaching 661 with `reject=0`. The evaluator reported:

- `ekf_vs_odom`: mean 0.041 m, P95 0.095 m, max 0.230 m
- `ekf_vs_aligned_gnss`: mean 0.484 m, P95 1.497 m, max 3.345 m

Strong GNSS display check used:

```bash
roslaunch ekf ekf_lidar.launch start_rviz:=true \
  rviz_config:=/home/zcl/catkin_ws/src/ekf/launch/data2_aligned_gnss_display.rviz \
```

Full replay with the same strong GNSS parameters reported:

- `reset=0`, `odom_realign=0`, `GNSS reject=0`

The latest machine-readable result is
`results/data2_aligned_gnss/data2_smooth_benchmark.json`.

## Important limitation

`/ekf/aligned_gnss/fix` is a derived debug/fusion input built using odom. It is
useful for inspecting GNSS participation and preventing frame-mismatch artifacts
in data2, but it is not an independent GNSS truth source for accuracy evaluation.
